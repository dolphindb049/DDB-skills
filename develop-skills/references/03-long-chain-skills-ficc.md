# 长链条 Skill 的开发：以 FICC 定价为例

长链条 Skill 和简单重复型 Skill 不一样。它不是一个小动作，而是一条完整业务链路。开发时最容易犯的错误，就是把整条链硬塞进一个 Skill 里，结果最后谁都复用不了。

## 1. 什么叫长链条 Skill

典型特征：

- 步骤多
- 上下游依赖明显
- 存在中间产物
- 需要多次联调
- 某些步骤可以被别的任务复用

FICC 定价就是典型例子：

```text
外部数据源 → 数据准备 → 建库建表 → 导入 → 曲线构建 → 定价 → 风险分析 → 可视化/报告
```

如果把这一整条链写成一个大 Skill，会出现三个问题：

1. 任一小改动都要动整份文档和整套脚本
2. 其他任务无法复用中间模块
3. Agent 很难判断到底该用其中哪一段能力

所以正确做法不是“写一个大全 Skill”，而是“先拆模块，再联调”。

## 2. 推荐拆分方式

以 FICC 定价为例，最小可维护拆分通常是：

- `prepare_data_for_pricing`
- `pricing`
- `ddb-visualization`
- `execute-dlang`

它们的职责应该清楚分开：

### 2.1 数据准备 Skill

负责：

- 拉原始数据
- 清洗
- 标准化字段
- 写入约定库表

### 2.2 定价 Skill

负责：

- 读取标准化后的表
- 构建曲线
- 执行定价
- 产出风险指标

### 2.3 可视化 Skill

负责：

- 读取结果表
- 生成图表和报告
- 输出 HTML 或其他交付物

### 2.4 执行 Skill

负责：

- 提供统一的 DDB 脚本执行入口
- 让前面几个 Skill 不必重复造执行能力

## 3. 模块化不是形式问题，而是复用问题

为什么不能合并成一个大 Skill？因为一旦合并，通用性就没了。

比如：

- 别的业务可能只想复用“数据准备”
- 有的场景只需要“定价计算”
- 可视化经常会独立演化
- 执行器几乎所有 DDB Skill 都能共用

如果这些都混在一份 Skill 里：

- 任何任务都必须带上不需要的部分
- 文档会越来越长
- 脚本之间耦合越来越强
- 变更成本越来越高

## 4. 开发顺序

长链条 Skill 的推荐顺序是：

```text
先拆模块 → 每个模块单独跑通 → 约定接口 → 再做联调
```

不能反过来。

如果上来就想“从头到尾一把通”，最后通常会陷入：

- 上游没稳定就改下游
- 下游出错又怀疑上游
- 所有脚本互相 hardcode
- 无法定位问题在哪一层

## 5. 联调时最重要的是接口契约

模块拆开之后，决定成败的不是“文档多不多”，而是输入输出契约清不清楚。

至少要明确：

- 库名、表名
- 关键字段名
- 字段类型
- 分区规则
- 时间字段格式
- 中间产物放哪里
- 哪一步是前置步骤

典型联调问题：

- 上游写的是 `bondCode`，下游读的是 `securityId`
- 上游输出到 `dfs://data_pricing_skill`，下游却从 `dfs://bond_pricing_v2` 读
- 上游按日期分区，下游假设按资产分区
- 上游还没导完数，下游已经开始算

这些问题不是靠大模型“想明白”解决的，而是靠接口约定和实际联调解决的。

## 6. 文档应该怎么组织

长链条 Skill 的文档，不应该把所有业务知识塞进一个 `SKILL.md`。

更推荐这样分：

```text
prepare_data_for_pricing/
├── SKILL.md
├── scripts/
├── references/
│   └── io-contract.md

pricing/
├── SKILL.md
├── scripts/
├── references/
│   └── assumptions.md
```

写法原则：

- `SKILL.md` 只讲职责边界、最小步骤、触发条件
- `references/` 放输入输出契约、字段说明、复杂案例
- `scripts/` 按顺序编号，保留可执行入口

## 7. 一个常见误区

误区是：

> “长链条任务很复杂，所以应该做成一个更大的 Skill。”

这在工程上是反的。

真正复杂的任务，更应该模块化。因为复杂任务的复杂性不是被“写进一个文档”消灭的，而是被“拆成稳定接口的多个模块”降低的。

## 8. 实际建议

做 FICC 这类长链条任务时，建议固定下面的节奏：

1. 先画出业务链路
2. 找到能独立验证的模块边界
3. 每个模块先单独跑通
4. 给模块补输入输出契约
5. 再按真实顺序联调
6. 联调通过后，再把踩坑补到 `references/`

顺序不能乱。

## 9. 结论

长链条 Skill 的核心不是“写得更全”，而是“拆得更对”。

以 FICC 定价为例，真正高效的做法是：

- 把数据准备、定价、可视化、执行器拆开
- 每个模块各自跑通
- 用明确契约做联调
- 最后用多个 Skill 组合成完整工作流

不要把整条链合成一个 Skill。那样看起来省事，实际上会同时失去：

- 模块化
- 通用性
- 可维护性
- 可复用性
