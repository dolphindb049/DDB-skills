# 简单重复型 Skill 的开发示例

这类 Skill 的特点是：任务本身不复杂，但重复很多次，而且步骤稳定、容易规范化。越是这种任务，做成 Skill 的收益越高。

## 1. 什么叫“简单重复型”

典型特征：

- 不是大产品，不需要复杂编排
- 依赖的外部系统有限
- 步骤相对固定
- 人工执行时很容易忘细节
- 做过两三次之后就明显烦了

这类任务非常适合用“实践一次，立刻归纳成 Skill”的方式沉淀。

## 2. 例子一：execute-dlang

`execute-dlang` 做的事情其实很聚焦：执行 DolphinDB 脚本或代码片段，并在需要时保留会话。

它为什么适合做 Skill：

- 用户会反复提“执行这段 DDB 代码”“跑一下这个 .dos 文件”
- 连接参数、执行方式、server 模式、PowerShell 引号坑都高度重复
- 一旦写成 Skill，Agent 就不需要每次临时猜命令

它的价值不在复杂，而在于把这些高频细节固定下来：

- 什么时候直接 `-c`
- 什么时候必须改成 `.dos`
- 什么时候要启 server 模式
- 返回值为空时怎么判断
- 路径、buffer、类型格式化有什么坑

这就是典型的“少量工具 + 很强复用”的 Skill。

## 3. 例子二：DolphinDB 部署

`ddb-deployment-skill` 也是很典型的简单重复型 Skill。

从工程角度看，部署 DDB 不是什么难算法，但它很容易出事故：

- 杀错进程
- 二进制名和脚本没改一致
- `localSite` 不匹配
- 高可用配置三台机器不一致
- license 复制错

这类任务一旦人工做，很依赖经验；但经验本身又很规范化，非常适合写成 Skill。

它的核心不是“帮你发明部署方案”，而是“把部署过程里的安全约束写死”。

## 4. 例子三：Mihomo 代理配置

`mihomo_onoff` 的价值更明显：

- 每台机器都要重复安装
- 每次都要处理路径、订阅、最小配置、启停命令
- 首次可用比花哨配置重要得多

这种任务如果不写成 Skill，团队里每个人都要重新踩一遍：

- 可执行文件放哪
- 配置文件放哪
- `PATH` 怎么配
- 为什么 GEOIP/MMDB 超时导致启动失败

写成 Skill 后，重点不是文笔，而是：

- 统一目录约定
- 提供最小可用配置
- 提供统一的 `on/off/status`
- 把常见失败模式直接写清楚

## 5. 这类 Skill 为什么开发简单

因为它们大多不需要“先设计一个体系”，而是直接按下面的闭环做：

```text
先做成一次 → 跑通 → 把对话里的步骤整理出来 → 变成 SKILL.md + scripts
```

这里最重要的一点是：**不要脱离真实执行去空写 Skill。**

正确做法：

1. 先在真实环境里做成一次
2. 中间遇到报错、兼容性、路径问题就现场修
3. 跑通后，把真实命令、真实路径、真实坑整理成文档
4. 把真正会用到的命令沉淀进 `scripts/`
5. 把 token、密码、地址这些敏感信息移到 `.env`

## 6. 推荐结构

简单重复型 Skill 不需要写得很大，结构越短越好：

```text
my-skill/
├── SKILL.md
├── scripts/
│   ├── 01_run.sh
│   └── 02_check.sh
├── references/
│   └── pitfalls.md
└── .env.example
```

推荐写法：

- `SKILL.md` 只放最小跑通路径
- `scripts/` 放直接可执行的命令或脚本
- `references/` 放坑和变体
- `.env.example` 放参数模板

## 7. 开发这种 Skill 的最实用方法

### 方法一：直接在会话里归纳

这是最高效的方式。

做完任务后，直接在当前对话里让强模型帮你输出：

- 这个 Skill 何时触发
- 最小操作步骤
- 常见坑
- 需要哪些脚本
- 哪些信息要抽到 `.env`

### 方法二：先脚本，后文档

如果你已经有可用命令，把命令先固化成脚本，再补 `SKILL.md`。

### 方法三：先文档，后补脚本

如果一开始命令还不稳定，可以先写 `SKILL.md` 的决策逻辑，等命令稳定后再补 `scripts/`。

## 8. 什么时候不该做成 Skill

下面这些场景不值得单独做 Skill：

- 只做一次的临时任务
- 没有稳定步骤的探索型任务
- 依赖太多隐含背景、别人几乎不会复用的私人脚本

判断标准很简单：

- 会不会再做第二次
- 别人会不会也需要
- 有没有大量可标准化细节

只要这三个问题里有两个答案是“会”，就值得沉淀。

## 9. 结论

简单重复型 Skill 的核心价值不是“复杂”，而是“稳定可复用”。

像 `execute-dlang`、DDB 部署、代理配置这类任务，本质上都是：

- 重复性强
- 容易出错
- 可规范化
- 一旦做成 Skill，团队收益非常大

所以这类 Skill 的正确开发方式不是长篇设计，而是：

**先做完，再总结；先跑通，再归档。**
